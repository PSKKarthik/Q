1. PRODUCT OVERVIEW
================================================================================

────────────────────────────────────────────────────────────────────────────────
1.1 PRODUCT NAME
────────────────────────────────────────────────────────────────────────────────

  QGX — Query Gen X
  Tagline: "The AI-powered academic platform built for modern learners."

────────────────────────────────────────────────────────────────────────────────
1.2 PROBLEM STATEMENT
────────────────────────────────────────────────────────────────────────────────

  Educational institutions rely on fragmented tools: one for tests, another for
  grading, another for communication, and none that connect them. Students lose
  engagement without feedback loops. Teachers waste hours on manual grading and
  formatting. Parents have zero visibility into daily academic performance.

  There is no single platform that combines:
  - AI-assisted content creation
  - Gamified learning incentives
  - Real-time communication between all stakeholders
  - Automated risk detection for struggling students

────────────────────────────────────────────────────────────────────────────────
1.3 TARGET USERS (PERSONAS)
────────────────────────────────────────────────────────────────────────────────

  PERSONA 1 — STUDENT (Primary User)
    Age:         14–22
    Context:     Enrolled in a school or coaching centre using QGX
    Goal:        Study, take tests, track grades, earn XP, get AI help
    Pain Point:  No consolidated view of marks, timetable, assignments, AI help
    Tech Level:  Medium-high; comfortable with mobile-first apps

  PERSONA 2 — TEACHER
    Age:         25–55
    Context:     Educator managing 20–100 students across subjects
    Goal:        Create and grade tests/assignments, track class performance
    Pain Point:  Manual grading, no plagiarism tools, no early-risk detection
    Tech Level:  Medium; needs intuitive UI for complex workflows

  PERSONA 3 — ADMIN
    Age:         30–60
    Context:     Academic coordinator or institution IT manager
    Goal:        Manage all users, announcements, platform health, analytics
    Pain Point:  No unified admin panel; can't see system-wide activity
    Tech Level:  Medium; needs control without requiring developer access

  PERSONA 4 — PARENT
    Age:         35–60
    Context:     Guardian of 1+ enrolled students
    Goal:        Monitor child's grades, attendance, communicate with teachers
    Pain Point:  No visibility into daily academic status; can't contact teachers
    Tech Level:  Low-medium; needs simple, child-scoped views

────────────────────────────────────────────────────────────────────────────────
1.4 CORE VALUE PROPOSITION
────────────────────────────────────────────────────────────────────────────────

  "One platform where students learn with AI assistance, teachers teach with
  data, admins govern with full visibility, and parents stay informed — all
  connected in real time."

  Key differentiators:
  1. AI-generated tests + AI tutor powered by Groq LLM
  2. Gamification (XP, levels, badges, quests, leaderboard) driving engagement
  3. End-to-end assessment: create → distribute → submit → grade → certify
  4. Predictive risk alerts for early intervention on struggling students
  5. Real-time messaging and forum with persistent chat threads

2.1 ROLE DEFINITIONS
────────────────────────────────────────────────────────────────────────────────

  Four roles exist in the system. Stored as: role = 'admin' | 'teacher' |
  'student' | 'parent' in the profiles table.

  ADMIN
    Description: Platform-wide governance. Oversees all content and users.
    Created by:  Existing admin via Create User modal only.
                 Cannot self-register via /register.
    Scope:       All data across all users.

  TEACHER
    Description: Content creator and student assessor.
    Created by:  Admin or self-registration via /register.
    Scope:       Own courses, tests, assignments; students on those items.

  STUDENT
    Description: Learner and test taker.
    Created by:  Admin or self-registration via /register.
    Scope:       Own data only (grades, attendance, submissions).

  PARENT
    Description: Observer linked to specific children.
    Created by:  Admin or self-registration via /register.
    Scope:       Linked children's data only. No academic content creation.

F-01: AUTHENTICATION & IDENTITY
────────────────────────────────────────────────────────────────────────────────

Description:
  Secure login using email/password OR QGX ID. Session managed via Supabase
  Auth with HTTP-only cookie. Middleware enforces role-based route access.

User Stories:
  - As a user, I can log in with my email and password.
  - As a user, I can log in using my QGX ID instead of email.
  - As a user, I am redirected to my role-specific dashboard after login.
  - As a user, a deep link with ?redirect=/dashboard/teacher is honored if my
    role matches.

Acceptance Criteria:
  AC-01  Login form accepts either email (contains @) or QGX ID (matches
         /^QGX-/i pattern).
  AC-02  QGX ID input resolves to the associated email by querying
         profiles.qgx_id = input.toUpperCase() before calling signInWithPassword.
  AC-03  Wrong password or non-existent QGX ID displays: "Invalid credentials".
         User remains on /login. No session is created.
  AC-04  On success, user is redirected to /dashboard/{role} where role is
         fetched from profiles.role.
  AC-05  If ?redirect= parameter is present AND starts with "/dashboard/" AND
         the role in the URL matches logged-in user's role, redirect to that URL.
         Otherwise, redirect to /dashboard/{role}.
  AC-06  Middleware protects all /dashboard/* routes. Unauthenticated requests
         are redirected to /login?redirect={original_path}.
  AC-07  Middleware checks profiles.role against URL segment. Mismatch redirects
         to /dashboard/{actual_role}.
  AC-08  VALID_ROLES = ['admin', 'teacher', 'student', 'parent']. Any other
         value in profile.role redirects to /login.
  AC-09  Logout clears session via supabase.auth.signOut() and redirects to
         /login.
  AC-10  Profile fetch failure in middleware redirects to /login (fail-safe).

In Scope:
  - Email + password login
  - QGX ID login (resolves via DB query)
  - Role-based routing post-login
  - Safe redirect parameter validation
  - Middleware route protection on /dashboard/*
  - Logout

Out of Scope:
  - OAuth providers (Google, GitHub, etc.)
  - Magic link login
  - Biometric / passkey authentication
  - Session timeout warnings / keep-me-alive prompts
  - Multi-device session management

F-02: USER REGISTRATION
────────────────────────────────────────────────────────────────────────────────

Description:
  Self-service account creation for teachers, students, and parents.
  Admin accounts cannot be self-registered.

User Stories:
  - As a new user, I can create an account by providing name, email, password,
    and selecting my role.
  - As a new user, I receive a unique QGX ID upon registration.

Acceptance Criteria:
  AC-01  /register accepts: name (required), email (required), password
         (required), phone (optional), role (required).
  AC-02  Role selector offers exactly: student, teacher, parent.
         Option "admin" does NOT appear.
  AC-03  Email must match standard email regex. Invalid format shows inline
         error. Form does not submit.
  AC-04  Password must be ≥8 characters AND contain ≥1 letter AND ≥1 number.
         Failure shows inline error per rule violated.
  AC-05  On submit, system calls supabase.auth.signUp(email, password).
  AC-06  System generates QGX ID via RPC generate_qgx_id(p_role).
         Fallback client-side generation: prefix by role (ADM/TCH/STU/PAR) +
         4-digit padded count + random 4-char suffix.
  AC-07  Profile row created with fields: name, email, role, phone, avatar
         (initials), qgx_id, xp=0, score=0, ghost_wins=0, joined=today.
  AC-08  If email already registered: error "Email already in use. Please log
         in." User stays on /register.
  AC-09  If email confirmation is enabled in Supabase: redirect to /login with
         message "Check your email".
  AC-10  If email confirmation is disabled (auto-confirm): redirect to
         /dashboard/{role}.
  AC-11  /register is accessible even when unauthenticated.

In Scope:
  - Self-registration for student, teacher, parent
  - QGX ID generation
  - Email + password validation
  - Profile row creation

Out of Scope:
  - Admin self-registration (Admins are created by existing admins only)
  - Invitation-based registration flow
  - Email domain whitelisting
  - Captcha / bot protection

F-03: PASSWORD RECOVERY
────────────────────────────────────────────────────────────────────────────────

Description:
  Email-based password reset flow via Supabase recovery link.

User Stories:
  - As a user, I can request a password reset email from /forgot-password.
  - As a user, clicking the recovery link lets me set a new password.

Acceptance Criteria:
  AC-01  /forgot-password has a single email input field.
  AC-02  Submitting a registered email calls resetPasswordForEmail(email,
         { redirectTo: '/reset-password' }).
  AC-03  Success: page shows "Reset link sent. Check your inbox." Email input
         is cleared.
  AC-04  Any email format is accepted without pre-verification (Supabase handles
         non-existent email silently).
  AC-05  /auth/callback receives ?code= parameter, calls exchangeCodeForSession,
         then redirects to ?next= value.
  AC-06  SAFE_PATHS whitelist = ['/reset-password', '/dashboard/student',
         '/dashboard/teacher', '/dashboard/parent', '/dashboard/admin',
         '/login']. Any ?next= value not in this list defaults to
         '/reset-password'.
  AC-07  If code exchange fails: redirect to /forgot-password?error=expired.
         Page shows "Link expired. Request a new one."
  AC-08  /reset-password requires a live PASSWORD_RECOVERY auth event (from the
         hash token). Without a valid token, the reset form does not appear.
  AC-09  New password on /reset-password: same rules as registration
         (≥8 chars, ≥1 letter, ≥1 number). Confirm field must match.
  AC-10  On successful reset: redirect to /login?reset=success. Login page
         shows "Password updated successfully."
  AC-11  Token exchange has a 12-second check before giving up.

In Scope:
  - Forgot password email submission
  - Auth callback code exchange
  - /reset-password with token verification
  - New password validation + confirmation

Out of Scope:
  - SMS-based reset
  - Security question recovery
  - Admin-forced password reset (separate admin feature)

F-07: ASSESSMENT ENGINE (TESTS & QUIZZES)
────────────────────────────────────────────────────────────────────────────────

Description:
  Teachers create assessments with multiple question types, configurable
  anti-cheat settings, and XP rewards. Students sit assessments in an
  enforced exam environment.

User Stories:
  - As a teacher, I can create a test with a mix of question types.
  - As a teacher, I can configure per-test anti-cheat rules.
  - As a student, I can attempt a test within the allowed attempt limit.
  - As a student, my answers are scored server-side and XP is awarded.

Acceptance Criteria:
  AC-01  Test form: title (required), subject (required), duration in minutes
         (required, >0), type ('test' | 'quiz'), scheduled_date, scheduled_time
         (both optional).
  AC-02  Supported question types per question row:
         mcq (1 correct), msq (multiple correct), tf (true/false),
         fib (fill-in-blank), match (left-right pairs).
  AC-03  Each question has: text, options (if applicable), correct answer,
         marks (>0), order_index.
  AC-04  Anti-cheat settings per test:
         tabSwitch (bool): Log/warn on tab switch.
         copyPaste (bool): Disable keyboard copy-paste shortcuts.
         randomQ (bool): Randomize question order.
         randomOpts (bool): Randomize option order within MCQ.
         fullscreen (bool): Enforce fullscreen on entry.
         timePerQ (int, seconds): Per-question countdown if >0.
         maxAttempts (int ≥1): Maximum submissions allowed.
         allowImmediateReview (bool): Show results immediately on submission.
         requireAllAnswered (bool): Block submit until all questions answered.
  AC-05  Default anti-cheat: all boolean flags = false, timePerQ = 0,
         maxAttempts = 1.
  AC-06  POST /api/submit-test validates: user is authenticated, role = student,
         test exists, student attempts < test.anti_cheat.maxAttempts.
         Violations return 403 with specific reason.
  AC-07  Scoring is performed server-side per question type:
         MCQ: answer_map[q.id] === q.answer → full marks.
         MSQ: sorted arrays match exactly → full marks, else 0.
         TF:  answer_map[q.id] === q.answer (boolean) → full marks.
         FIB: trimmed, case-insensitive string match → full marks.
         Match: each pair.right trimmed lowercase === answer pair.right
                trimmed lowercase → full marks. One wrong pair = 0 total.
  AC-08  XP awarded = min(Math.round((percent / 100) * test.xp_reward),
         MAX_XP_PER_TEST). MAX_XP_PER_TEST = 500.
  AC-09  If double XP is active: xp_earned * 2 (still capped at
         MAX_XP_PER_TEST constants-wise; hard cap at 500, or 1000 during 2x).
  AC-10  XP is applied via atomic RPC atomic_xp_update (prevents race
         condition on concurrent submissions).
  AC-11  Ghost scoring: student's percent compared to stored profile.score
         (best ever score). If new > best, ghost_wins incremented by 1.
  AC-12  Test deadline: If test has scheduled_date + scheduled_time → server
         computes deadline as scheduled start + duration. Submission after
         deadline returns 403 "Deadline passed".
  AC-13  Question ordering: If randomQ=true, questions shuffled via
         Fisher-Yates algorithm before presenting to student.
  AC-14  Attempt stored with: student_id, test_id, score, total, percent,
         answer_map (full map), submitted_at.
  AC-15  Student can review their answers post-submission only if
         allowImmediateReview = true.
  AC-16  XP per test reward is set by teacher at test creation (0–500).

In Scope:
  - Test/quiz creation by teachers
  - 5 question types (MCQ, MSQ, TF, FIB, Match)
  - Anti-cheat configuration
  - Server-side scoring
  - XP reward on submission
  - Double XP multiplier
  - Attempt tracking with full answer map
  - Deadline enforcement

Out of Scope:
  - Partial credit for MSQ
  - Image/video questions
  - Audio questions
  - Peer-reviewed grading
  - Proctoring (camera/screen recording)
  - Question banks across tests
  - Weighted question difficulty

F-08: AI QUESTION GENERATION
────────────────────────────────────────────────────────────────────────────────

Description:
  Teachers upload a document and the system generates structured questions
  using the Groq LLM API.

User Stories:
  - As a teacher, I can upload a PDF or document and receive generated questions.
  - As a teacher, I can review, edit, and import generated questions into a test.

Acceptance Criteria:
  AC-01  POST /api/ai accepts: file upload OR text prompt.
  AC-02  Supported file types: PDF, PPT/PPTX, images (jpg, png, webp).
  AC-03  Maximum file size for AI upload: 5MB (distinct from course file 50MB
         limit).
  AC-04  Rate limit: 10 requests per minute per authenticated user.
         Exceeding returns 429 "Rate limit exceeded. Retry after {N}s."
  AC-05  Role check: Student role → tutor mode. Teacher role → question
         generation mode.
  AC-06  Generated questions are returned as a JSON array. Each item must
         conform to Question interface: type, text, options (if MCQ/MSQ),
         answer, marks.
  AC-07  Teacher reviews generated questions in a preview panel before
         importing to the active test.
  AC-08  JSON extraction from LLM response: content between first '[' and last
         ']' is parsed as array. Invalid JSON returns error toast.
  AC-09  In-memory rate limiter resets on process restart. Noted as dev-only
         behavior (non-persistent).

In Scope:
  - PDF content extraction → question generation
  - Question preview + selective import
  - Rate limiting per user
  - Question type: MCQ, MSQ, TF, FIB, Match

Out of Scope:
  - Video/audio document upload for AI

F-23: PLAGIARISM DETECTION
────────────────────────────────────────────────────────────────────────────────

Description:
  Teacher-facing tool that scans all assignment submissions for text
  similarity using n-gram analysis and flags suspicious pairs.

User Stories:
  - As a teacher, I can scan an assignment's submissions for plagiarism.
  - As a teacher, I can set a similarity threshold and flag suspicious pairs.

Acceptance Criteria:
  AC-01  Algorithm: 4-word n-gram comparison between all submission pairs.
         Similarity = (shared n-grams / min(A ngrams, B ngrams)) * 100.
  AC-02  Minimum text length: 5 tokens. Shorter submissions return 0%
         similarity.
  AC-03  Tokenization: lowercase, strip non-alphanumeric, filter tokens <3
         characters.
  AC-04  Threshold: user-adjustable 10–90%. Default: 30%.
         Only pairs above threshold shown.
  AC-05  Color coding: ≥70% = red, ≥50% = orange, <50% = blue.
  AC-06  Shared phrases: up to 5 shared 4-grams per pair, highlighted in
         results.
  AC-07  Scan is limited to first 100 submissions. Warning shown if capped.
  AC-08  Flag statuses: 'open' | 'reviewed' | 'dismissed'. Teacher can update
         status per flag.
  AC-09  Scope: only teacher's own assignments and their submissions.

In Scope:
  - 4-gram n-gram similarity
  - Adjustable threshold
  - Per-pair visualization with shared phrases
  - Flagging workflow (open, reviewed, dismissed)
  - Teacher-scoped results

Out of Scope:
  - Cross-internet plagiarism (Turnitin-style)
  - Code plagiarism detection
  - Image similarity
  - Plagiarism report export

F-24: PREDICTIVE RISK ALERTS
────────────────────────────────────────────────────────────────────────────────

Description:
  Automated system that analyzes student data and produces a risk score
  (0–100) to help teachers identify students needing intervention.

User Stories:
  - As a teacher, I see a prioritized list of at-risk students with flags.

Acceptance Criteria:
  AC-01  Risk score components (max 100):
         - Attendance: <70% = +30, 70-85% = +15, ≥85% = 0.
         - Test avg: <40% = +30, 40-60% = +15, ≥60% = 0.
         - Performance trend: If recent 3 tests avg < prior 3 avg by >10% = +15.
         - Missed assignments: ≥3 overdue = +20, 1-2 = +10, 0 = 0.
         - Inactivity: No attempts in last 14 days = +10.
  AC-02  Risk levels: High ≥40, Medium 20-39, Low <20.
  AC-03  Flags generated per student listing specific reasons with values,
         e.g. "Low attendance (62%)", "3 missed assignment(s)".
  AC-04  Trend indicator per student: ▼ Declining | ▲ Improving | — Stable.
         Calculated from last 3 vs prior 3 test attempts.
  AC-05  Scope: Teacher sees only students who have attempted teacher's tests
         or submitted to teacher's assignments.
  AC-06  List sorted by risk score descending. Paginated (PAGE_SIZE = 20).
  AC-07  Manual refresh button triggers full data recompute.

In Scope:
  - 5-component risk scoring
  - Specific risk flags with values
  - Trend direction
  - Teacher-scoped view
  - Manual refresh

Out of Scope:
  - Automated alerts/emails sent to parents
  - ML model-based prediction
  - Risk history over time
  - Admin-wide risk view

4. FUNCTIONAL REQUIREMENTS
================================================================================

FR-01  All /dashboard/* routes are protected by Next.js middleware. No
       unauthenticated or mismatched-role access is possible.

FR-02  Role is stored in profiles.role and is the authoritative source.
       Session token alone is insufficient; profile must be queryable.

FR-03  All destructive actions (delete user, delete course, delete post)
       require explicit confirmation from the UI before executing.

FR-04  All data-mutating API routes verify authentication via
       supabase.auth.getUser(). No action proceeds without a valid user.

FR-05  All data-mutating API routes re-verify the caller's role from the
       profiles table server-side. Client-supplied role claims are ignored.

FR-06  The system generates QGX IDs for all users.
       Format: QGX-{PREFIX}{NNNN}-{RAND4} where PREFIX = ADM/TCH/STU/PAR.

FR-07  XP updates use atomic database RPC (atomic_xp_update). Client-side
       XP arithmetic is for display only, not for persistence.

FR-08  File uploads have a hard size limit of MAX_FILE_SIZE = 50MB enforced
       client-side before upload. API-level file processing (AI) enforces 5MB.

FR-09  Open redirect attacks are prevented. The ?redirect= parameter on
       /login must start with "/dashboard/". The ?next= parameter on
       /auth/callback must match the SAFE_PATHS whitelist exactly.

FR-10  All user-generated content displayed in HTML (forum posts, messages)
       is sanitized: HTML entities escaped, only https:// links rendered,
       javascript: URIs blocked.

FR-11  Pagination is consistent: PAGE_SIZE = 20 records per page across all
       lists (users, forum posts, alerts, notifications).

FR-12  CSV export is available on: grades, attendance records, analytics
       data, assignment submissions. Generated client-side.

FR-13  Debounce of 300ms (DEBOUNCE_MS) applied to all search input fields
       before triggering DB queries.

FR-14  Activity logging: All admin-initiated user operations (create, delete)
       are recorded in the activity_log table with actor_id and message.

FR-15  Error boundaries exist at the app level (app/error.tsx). Unhandled
       module-level errors display the error page with "Try Again" and
       "Home" actions.

6. EDGE CASES & ERROR HANDLING
================================================================================

EC-01  LOGIN — Invalid credentials
       → Response: 401. UI message: "Invalid credentials". No session.
       User stays on /login.

EC-02  LOGIN — QGX ID not found in profiles table
       → Profile query returns null. Fallback attempt with raw input fails.
       UI message: "Invalid credentials".

EC-03  MIDDLEWARE — Profile fetch fails (DB error)
       → try/catch in middleware. Redirect to /login for safety.
       No dashboard access without confirmed profile.

EC-04  MIDDLEWARE — role in profile is not a valid Role value
       → VALID_ROLES check fails. Redirect to /login.

EC-05  REGISTRATION — Duplicate email
       → Supabase returns error. UI message: "Email already in use."
       User stays on /register.

EC-06  AUTH CALLBACK — Expired or invalid code
       → exchangeCodeForSession fails. Redirect to /forgot-password?error=expired.
       UI message: "Link expired. Please request a new one."

EC-07  TEST SUBMISSION — Student exceeds max attempts
       → POST /api/submit-test returns 403 "Maximum attempts exceeded".
       Student sees test as completed, no "Start" button shown.

EC-08  TEST SUBMISSION — Submission after deadline
       → Server calculates deadline = scheduledStart + duration.
       Returns 403 "Deadline passed".

EC-09  TEST SUBMISSION — Non-student role
       → Returns 403 "Access denied". No score or XP recorded.

EC-10  USER DELETION — Self-deletion attempt
       → POST /api/delete-user returns 403 "Cannot delete your own account".

EC-11  USER DELETION — Deleting last admin
       → Returns 403 "Cannot delete the last administrator".

EC-12  USER DELETION — User ID does not exist
       → Supabase delete succeeds silently (no-op). No error returned.
       [KNOWN GAP — not validated before deletion attempt]

EC-13  AI API — Rate limit exceeded
       → Returns 429 "Rate limit exceeded. Retry after Ns."

EC-14  AI API — File too large (>5MB)
       → Returns 400 "File too large".

EC-15  AI API — Invalid file type
       → Returns 400 with file type rejection message.

EC-16  AI API — LLM returns malformed JSON (no array found)
       → JSON extraction fails. Returns 500 with parsing error.

EC-17  FILE UPLOAD — File exceeds 50MB
       → Client-side check (MAX_FILE_SIZE). File rejected before upload.
       Toast: "File too large. Maximum 50MB."

EC-18  COURSE COMPLETION — No course files exist
       → Certificate never issues (no files to complete).
       Student sees 0% progress, no claim button.

EC-19  PARENT LINKING — Invalid QGX ID
       → Profile query returns null. UI message: "Student not found."
       No parent_students row created.

EC-20  PARENT DATA VIEW — No children linked
       → All Monitor tabs show empty state with "Link a student" CTA.
       No data until at least one child is linked.

EC-21  PASSWORD RESET — /reset-password without token
       → PASSWORD_RECOVERY event never fires. Reset form not shown.
       User sees waiting state or is prompted to use forgot-password.

EC-22  PLAGIARISM SCAN — More than 100 submissions
       → Only first 100 processed. Warning toast: "Showing first 100 submissions."

EC-23  CODE PLAYGROUND — Infinite loop (JavaScript)
       → 5-second timeout in Web Worker. Displays: "Execution timed out"
       in output panel. Worker terminated and restarted.

EC-24  CODE PLAYGROUND — Unsupported Python syntax (def, class, import)
       → Custom error: "Unsupported syntax. Python mode supports basic
       statements only."

EC-25  QUEST CLAIM — Already claimed
       → claimed = true. Claim button disabled. XP not double-awarded.

EC-26  FORUM ATTACHMENT — File exceeds 10MB
       → Client-side check. Rejected before upload.

EC-27  DAILY LOGIN BONUS — Already claimed today
       → RPC claim_daily_login_xp returns error. Toast: "Already claimed
       today." No duplicate XP.

EC-28  REALTIME — WebSocket connection drops
       → Supabase client handles reconnection automatically. UI may show
       stale data briefly until reconnect.

EC-29  BUILD — Missing environment variables
       → lib/supabase.ts throws on startup: "Missing Supabase environment
       variables: NEXT_PUBLIC_SUPABASE_URL and NEXT_PUBLIC_SUPABASE_ANON_KEY".

EC-30  GLOBAL ERROR — Any unhandled page-level exception
       → app/error.tsx renders. Shows: error message, "Try Again" (reloads
       component), "Home" (navigates to /).
